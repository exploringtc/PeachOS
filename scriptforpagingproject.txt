PeachOS Long Mode + Four-Level Paging Project Script

Author Context
- Base repository branch: main from exploringtc/PeachOS
- Working branch/result: feature/long-mode-paging
- Goal: Convert the original 32-bit kernel path into a verifiable 64-bit long-mode bootstrap with working 4-level paging and hardware-level proof in QEMU + GDB.

============================================================
1) Project Objective (What this project achieves)
============================================================
This project demonstrates a correct x86_64 architectural transition from 32-bit protected mode into 64-bit long mode, then verifies the result using control-register and memory inspection rather than output-only behavior.

Target outcomes achieved:
1. Enter 64-bit long mode successfully.
2. Use a valid 64-bit GDT.
3. Enable paging correctly with CR0 and CR4.
4. Load a valid PML4 base into CR3.
5. Implement a functioning 4-level hierarchy: PML4 -> PDPT -> PD -> PT.
6. Boot stably under QEMU.
7. Prove correctness through GDB register and memory evidence.

============================================================
2) High-Level Design Choices (Why this implementation looks this way)
============================================================
Design decision #1: Minimize moving parts during architecture migration.
- We intentionally reduced the active build path to the smallest set required for long-mode and paging verification.
- Rationale: trying to carry every legacy subsystem (filesystem/tasking/interrupt-heavy flows) through an architecture jump often hides core CPU-mode bugs behind unrelated module failures.

Design decision #2: Prioritize hardware-state evidence over cosmetic output.
- Kernel printouts are useful but not sufficient.
- We relied on GDB proof points (RIP, CR0, CR4, CR3, and PML4 dump), which directly satisfy grading requirements.

Design decision #3: Keep early paging identity-mapped and simple.
- The first 2 MiB is identity-mapped with 4 KiB pages.
- Rationale: this keeps instruction/data addresses valid immediately after enabling paging and makes failure analysis straightforward.

7) Address-Translation Explanation (for report narrative)
============================================================
In x86_64 long mode, a virtual address is translated by walking four tables:
1) PML4
2) PDPT
3) PD
4) PT
Then the page offset is appended to produce the final physical address.

This project configures the first valid chain:
- PML4[0] -> PDPT
- PDPT[0] -> PD
- PD[0] -> PT
- PT[i] -> physical frame i * 0x1000

Because the mapping is identity for the first 2 MiB, code and data addresses remain valid immediately after paging is enabled, simplifying early boot stability.

8) Challenges Encountered and How They Were Resolved
============================================================
Challenge 1: Architecture mismatch risk during migration.
- Issue: Legacy 32-bit assumptions in toolchain/object format can cause boot and linkage failures.
- Resolution: Unified toolchain and object formats around x86_64-elf + elf64.

Challenge 2: Debugging noise from unrelated legacy modules.
- Issue: Filesystem/tasking/interrupt setup can fail independently and obscure long-mode root causes.
- Resolution: Reduced active path to minimal long-mode demonstration kernel.

Challenge 3: Ensuring proof is grader-verifiable.
- Issue: Printed output alone is not enough for rubric acceptance.
- Resolution: Captured register and memory evidence via GDB (CR0/CR3/CR4 + PML4 dump).
